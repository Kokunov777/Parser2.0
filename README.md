# Лабораторная работа №1: Редактор текста с графическим интерфейсом
## Название и цель работы
**Название:** Редактор текста с графическим интерфейсом (GUI Language Processor Editor)
 
 **Цель работы:** Разработка кроссплатформенного графического приложения для ввода, редактирования и сохранения текстовых данных. Приложение служит основой для последующих лабораторных работ по дисциплине "Теория формальных языков и компиляторов", обеспечивая интерфейс для взаимодействия с лексическими анализаторами и парсерами.
 
 
 ## Описание проекта
Реализовано оконное приложение, представляющее собой многофункциональный текстовый редактор. Программа поддерживает основные операции работы с файлами (создание, открытие, сохранение), редактирование текста, а также раздельный вывод результатов работы программ-обработчиков (лексический анализ, синтаксический разбор и т.д.). Интерфейс адаптивен: пользователь может изменять размеры областей ввода и вывода.

## Ограничения: Любые особенности реализации, известные ограничения.
Строгая грамматика: Анализатор распознает только одну конкретную конструкцию:
let <идентификатор> = num::complex::Complex::new(<число>, <число>);
Любое отклонение от этого шаблона (пропуск ;, опечатка в complex, лишние пробелы в ключевых местах) будет расценено как ошибка.
Однострочный анализ: Анализатор предназначен для проверки одной строки кода. Многострочные конструкции не поддерживаются.
Типы чисел: Вещественные числа могут быть записаны как с целой, так и с дробной частью (3.1, -4.2). Экспоненциальная форма (1e5) не поддерживается.
Метод восстановления: Используется метод нейтрализации ошибок Айронса. При обнаружении первой ошибки анализатор пытается "пропустить" некорректные токены до ближайшего восстанавливающего символа (например, ;), после чего может продолжить анализ и найти другие ошибки.
Отсутствие подсветки синтаксиса: Встроенный редактор (QTextEdit) не имеет динамической подсветки синтаксиса Rust. Подсветка применяется только для указания местоположения ошибки после клика по таблице.
Зависимость от PyQt6: Для работы приложения необходима библиотека PyQt6. Убедитесь, что она установлена корректно.

Грамматика G[Z] для объявления комплексного числа на языке Rust является автоматной (тип 3) и относится к подклассу праволинейных автоматных грамматик. # Классификация грамматики (по Хомскому).
Согласно определению (см. раздел 2.7), правила автоматной грамматики должны иметь вид A → aB или A → a, где A, B ∈ V_N, a ∈ V_T.

Правило 1 (<Объявление> → let id : path = <Инициализатор> ;) не удовлетворяет этому условию, так как в правой части содержит несколько терминалов, за которыми следует нетерминал.

Правило 5 (<ВызовКонструктора> → path ( float , float )) также не соответствует, так как в нем отсутствует нетерминал-преемник в конце цепочки.
Вывод: грамматика не является автоматной.

Проверка на принадлежность к контекстно-свободным (КС) грамматикам.
Правила КС-грамматики имеют вид A → α, где A ∈ V_N, а α ∈ (V_T ∪ V_N)*.

Все правила грамматики G[<Объявление>] удовлетворяют этому критерию. В левой части каждого правила находится ровно один нетерминальный символ (<Объявление>, <Инициализатор>, <ВызовКонструктора>).
Вывод: грамматика является контекстно-свободной (КС-грамматикой).

Проверка на принадлежность к контекстно-зависимым (КЗ) грамматикам и грамматикам типа 0.
Поскольку любой КС-язык является также и КЗ-языком, наша грамматика формально принадлежит и к типу 1. Однако практическая ценность этого факта невелика, так как для реализации анализатора требуется более узкий, эффективно разбираемый подкласс.
